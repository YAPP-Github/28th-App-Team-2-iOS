# GitHub Issue Management Guidelines

이 프로젝트의 진척 상황을 유연하게 추적하고 관리하기 위해 이슈 생성 및 설계 시 다음 지침을 반드시 준수한다. 모든 이슈 조작은 로컬 `gh` CLI 도구를 사용한다.

## 1. 하위 이슈 (Sub-issue) 분리 원칙
- **작업 크기 제어**: 사용자가 지시한 요구사항이 2일 이상 소요되거나 다수의 모듈/피처를 동시에 건드려야 하는 큰 작업(예: "운세 화면 전체 개발")일 경우, 이를 1~2일 내에 완료 및 리뷰 가능한 세부 단위로 쪼개어 개별 이슈로 나눈다.
- **부모-자식 이슈 트래킹 프로세스**:
  1. **부모 이슈 생성**: 먼저 전체 대형 작업을 기술하는 최상위 부모 이슈(Epic)를 생성한다. 부모 이슈는 반드시 `[Epic]` 타입으로 생성한다.
     * 예시 제목: `[iOS] [Epic] [온보딩] 온보딩 화면 전체 구현`
  2. **Tasklist 작성**: 부모 이슈 본문에 대괄호 체크박스 목록(`- [ ] 세부작업`)을 사용하여 세부 태스크를 나열한다.
  3. **자식 이슈 생성 및 연결 (CLI 자동화)**: 나열된 각 체크박스 작업을 자식 이슈로 생성할 때, 반드시 **`gh issue create --parent [부모이슈번호]`** 옵션을 지정하여 부모 이슈와 자식 이슈 간의 추적 관계(Tracked in)가 깃허브 상에 명확히 연결되도록 조치한다.
- **분리 예시**:
  * 부모 이슈: `[iOS] [Epic] [운세] 운세 화면 전체 구현` (이슈 번호: #100)
  * 하위 이슈 1: `[iOS] [Feat] [운세] 운세 화면 UI 레이아웃 구현` (명령어: `gh issue create --parent 100 ...`)
  * 하위 이슈 2: `[iOS] [Feat] [운세] 운세 화면 API 연동 및 네트워크 레이어 구성` (명령어: `gh issue create --parent 100 ...`)

## 2. 이슈 생성 및 라벨/보드 연동 규칙
- **이슈 제목 규칙 (Scoped Type)**: `[iOS] [타입] [화면/도메인명] 상세 태스크 내용` 형태로 일관되게 생성한다. (타 직군과의 협업 보드 내 식별을 위해 **`[iOS]`**를 최상단 헤더에 필수 기재한다.)
  - **허용되는 [타입]**: `[Epic]`, `[Feat]`, `[Bug]`, `[Docs]`, `[Refactor]`, `[Chore]`
  - **허용되는 [화면/도메인명]**: 
    * 화면 도메인: `[온보딩]`, `[운세]`, `[행운액션]`, `[토닥이]`, `[마이페이지]`
    * 공통 도메인: `[Core]`, `[App]`, `[Tuist]`
  - **도메인 생략 예외 규칙**: 프로젝트 전체 설정, 에이전트 툴링 설정, CI/CD 스크립트 수정 등 특정 화면이나 공통 레이어 모듈 도메인에 명확히 속하지 않는 순수 설정/인프라 작업의 경우 `[화면/도메인명]`을 완전히 생략하고 작성한다.
    * 예시: `[iOS] [Chore] AI 협업 가이드라인(Harness) 구축` (도메인 생략)
    * 예시: `[iOS] [Feat] [운세] 운세 UI 레이아웃 구현` (도메인 적용)
- **이슈 템플릿 준수 및 본문 생성**:
  - 이슈 생성 시 로컬 저장소 내에 보관된 **`.github/ISSUE_TEMPLATE/`** 폴더 하위의 템플릿 마크다운 문서를 기반으로 본문을 빌드한다:
    * **기능 개발/설정/기타 작업**: `task.md` 템플릿의 본문 서식을 채워 사용하고, 라벨은 **`enhancement`** 또는 **`documentation`**을 지정한다. (작업의 성격에 따라 두 라벨을 모두 지정하는 것도 가능하다.)
    * **버그 수정 작업**: `bug.md` 템플릿의 본문 서식을 채워 사용하고, 라벨은 **`bug`**를 지정한다.
  - 작성된 본문 텍스트를 임시 파일로 만들어 **`--body-file`** 플래그를 통해 생성하며, 이때 프로젝트 카드 지정(`--project "Todak"`) 및 담당자 지정(`--assignee`) 플래그를 함께 엮어 발행한다.
    * **명령어 예시**: `gh issue create --title "[iOS] [Feat] [운세] 운세 UI 구현" --body-file BODY.md --project "Todak" --label "enhancement" --assignee "@me"`
- **Page 대분류 필드 즉시 연동 규칙**:
  - 이슈가 생성되어 프로젝트 보드에 카드로 추가되면, AI는 **이슈 생성 즉시 해당 카드의 Page 필드 값을 변경**한다. (단, `Status` 필드는 건드리지 않고 기본값 `작업 이전` 상태로 둔다.)
  - **화면 외 공통 작업 예외 (Blank 허용)**: 특정 화면 도메인에 종속되지 않는 공통 작업(예: Core 모듈 설계, Tuist 설정, 단순 문서 갱신 등)의 경우, Page 필드 설정을 생략하고 공백(Blank)으로 유지한다.
  - **연동 흐름**:
    1. `gh` 명령어로 프로젝트 보드 내 생성된 이슈 카드의 ID(`itemId`)를 찾는다. (Owner: `YAPP-Github`, Project: `34`)
    2. `gh` 명령어로 프로젝트 보드의 필드 목록을 조회하여 `Page` 필드 ID와 해당 도메인에 부합하는 Option ID를 런타임에 찾는다.
    3. 찾은 ID를 기반으로 `gh project item-edit`를 사용해 `Page` 필드를 업데이트한다. (사용자 승인 필요)
- **담당자(Assignee) 지정**:
  - 이슈 생성 시 기본값으로 작업을 수행하는 주체인 **`@me`**를 배정하되, 타인에게 작업을 배정하거나 명시적인 지정 요청이 있는 경우 해당 깃허브 계정명을 지정하여 발행한다.
