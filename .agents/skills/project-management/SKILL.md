# Project Management Guidelines (Issue-Driven Development)

이 프로젝트에서 이루어지는 모든 기능 개발, 버그 수정, 리팩토링 등의 작업은 협업과 진척도 관리를 위해 **GitHub Projects (소유자: "YAPP-Github", 번호: 34, 이름: "Todak")**와 항상 동기화되어야 한다. **이 가이드라인 하에 진행되는 모든 GitHub 이슈 조회/생성 및 프로젝트 보드 제어는 로컬에 로그인 인증이 완료된 GitHub CLI (`gh` 도구)를 활용하여 수행한다.**

## 1. 작업 개시 (Issue & Project Board)
- **이슈 식별 규칙 (토큰 절약 및 Latency 방지)**:
  사용자가 새로운 작업을 요청할 때, AI는 무작정 깃허브 API 검색(`gh issue list`)을 돌려 토큰을 낭비하지 않고, 아래의 **3단계 우선순위**에 따라 이슈를 파악한다.
  1. **1단계 (직접 매핑)**: 사용자의 지시문 내에 이슈 번호가 이미 포함되어 있는 경우 (예: `"#12 온보딩 개발해줘"`)
     * ➡️ 별도의 검색이나 질문 없이 해당 이슈 번호를 즉시 매핑하고 개발을 준비한다.
  2. **2단계 (질문 우선)**: 이슈 번호가 식별되지 않는 일반적인 경우
     * ➡️ API 검색을 실행하지 않고, 사용자에게 먼저 질문을 던져 결정을 확인한다.
     * ➡️ 질문 템플릿: `"이 작업과 관련된 기존 깃허브 이슈 번호가 있나요? 없다면 새로 생성할까요, 아니면 이슈 없이 바로 진행할까요?"`
     * ➡️ 사용자가 이슈를 생성해달라고 동의하면 [issue-management](../issue-management/SKILL.md) 가이드를 준수하여 생성하고, 이슈 없이 가자고 하면 즉시 모든 프로젝트 보드 연동 단계를 건너뛴다.
  3. **3단계 (제한적 검색)**: 사용자가 "이슈를 검색해서 찾아달라"고 명시적으로 위임하거나, AI의 탐색이 반드시 필요하다고 합의된 제한적인 경우
     * ➡️ 이 경우에 한해서만 `gh issue list`를 돌려 매칭되는 이슈를 찾는다.

- **이슈 생성 전 최종 컨펌 의무화 (자의적 생성 금지)**:
  - `gh issue create` 명령어를 실행하기 바로 직전에, AI는 대화 창에 실제로 생성될 이슈 제목과 이슈 본문(Body) 초안을 마크다운 형식으로 사용자에게 반드시 먼저 보여준다.
  - 사용자로부터 "이대로 생성해줘" 또는 이에 동의하는 명시적인 답변을 얻은 직후에만 실제 생성 명령어를 실행한다.

- **프로젝트 보드 상태 갱신**: 
  - **Status 업데이트 타이밍**: 이슈를 새로 생성하거나 매칭되었다고 해서 **즉시 `작업 중`으로 변경하지 않는다.** 이슈 생성 시점에는 기본값인 `작업 이전` 상태를 유지한다.
  - 사용자가 해당 이슈를 대상으로 **실제 개발을 착수하도록 구체적으로 지시하는 시점**에 비로소 아래 규칙에 따라 프로젝트 보드 카드를 `작업 중`으로 변경한다. (이슈 없이 진행 시 생략)
  - **Status 업데이트 흐름 (CLI 가이드)**:
    1. `gh` 명령어를 통해 프로젝트 보드 내 해당 이슈 카드의 ID(`itemId`)를 찾는다. (Owner: `YAPP-Github`, Project: `34`)
    2. `gh` 명령어로 프로젝트 보드의 필드 목록을 읽어, Status 필드 내 **`작업 중`** 한글 명칭에 대응하는 Option ID를 런타임에 동적으로 찾아낸다.
    3. 동적으로 확보한 Option ID를 활용하여 해당 카드의 `Status`를 `작업 중`으로 변경한다. (사용자 승인 필요)

## 2. 코드 작업 연동 (Branch & Commit)
- 상세 브랜치 생성 및 커밋 컨벤션은 [git-harness](../git-harness/SKILL.md) 스킬의 지침을 따르되, 이슈 번호를 식별자로 매핑한다. (이슈 없이 진행 시 이슈 번호 매핑 단계는 생략)

## 3. 작업 마무리 (Pull Request & Close)
- **PR 생성 전 최종 컨펌 의무화 (자의적 생성 금지)**:
  - `gh pr create` 명령어를 실행하기 바로 직전에, AI는 대화 창에 실제로 생성될 PR 제목과 PR 본문 초안을 마크다운 형식으로 사용자에게 반드시 먼저 보여준다.
  - 사용자로부터 "이대로 생성해줘" 또는 이에 동의하는 명시적인 답변을 얻은 직후에만 실제 생성 명령어를 실행한다.

- **PR 생성 의사 확인 질문**:
  - 작업이 모두 완료되어 커밋 및 푸시가 끝나면, AI는 사용자에게 다음과 같이 질문하여 PR 작성 여부를 확인한다.
    > "작업이 완료되어 푸시되었습니다. GitHub PR을 생성할까요? 아니면 PR 없이 이대로 마무리할까요?"
  - **PR 생성 동의 시**: `gh pr create` 명령을 활용하여 PR을 생성하되, 구체적인 PR 템플릿 채우기 및 작성 방식은 [pull-request-management](../pull-request-management/SKILL.md) 가이드라인을 엄격히 준수한다. (PR 본문에 `Closes #이슈번호` 등의 자동 닫기 키워드 필수 주입)
  - **PR 미생성 시**: PR 작성을 생략하고 작업을 바로 종료한다.
- **자동화 위임**: 닫기 키워드를 통해 PR이 머지(Merge)될 때 프로젝트 보드 내 이슈 카드가 자동으로 `작업 완료` 상태로 이동되고 이슈가 닫히므로, AI가 CLI를 통해 직접 보드 상태를 강제 수정하지 않는다. (이슈 없이 진행 시 자동화 및 조작 생략)
