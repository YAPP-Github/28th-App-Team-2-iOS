# Git Harness Guidelines

이 프로젝트 공간에서 커밋 메시지를 작성하거나 Git 관련 작업을 수행할 때는 다음 규칙을 엄격히 준수한다.

## 1. 커밋 메시지 컨벤션
- **언어**: 커밋 메시지는 반드시 **한국어**로 작성한다.
- **형식**: Conventional Commits 포맷을 준수한다.
  - 타입 종류: `feat`(기능), `fix`(버그), `docs`(문서), `chore`(설정/기타), `refactor`(리팩토링), `test`(테스트)
  - 작성 예시: `feat(module): 회원가입 API 연동 (#12)`
- **이슈 번호 매핑**: 커밋 메시지 제목(Header) 끝에 매핑된 깃허브 이슈 번호를 괄호 형식으로 추가한다. (단, 이슈 없이 진행하기로 협의된 예외적인 경우엔 이슈 번호 매핑을 생략한다.)
- **상세 내용**: 본문(Body)에는 `- ` 로 시작하는 Bullet point를 사용하여 수정된 이유와 변경 사항을 자세하게 서술한다.
- **Co-author 등록 금지 (매우 중요)**: AI(Claude, Cursor 등)가 커밋 메시지를 대신 작성할 때, 본문 하단에 `Co-authored-by: AI <...>` 형태의 꼬리말을 달아 자신을 공동 작업자로 등록하는 행위를 절대 금지한다.

## 2. 브랜치 규칙
- 신규 브랜치 생성 시 Git Flow 규칙을 엄격히 따른다.
- 허용되는 **신규 브랜치 접두사**: `feature/`, `bugfix/`, `hotfix/`, `release/`, `private/`
- **이슈 번호 매핑**: 신규 브랜치를 생성할 때 접두사 바로 뒤에 이슈 번호를 포함시켜 브랜치의 소유와 역할을 명확히 한다. 특수문자 `#`은 브랜치명에서 제외하고 대시(`-`)로 구분한다. (예시: `feature/12-login-ui`, `bugfix/45-network-error`) (단, 이슈 없이 진행하기로 협의된 예외적인 경우엔 이슈 번호 기재를 생략한다.)
- `main`과 `develop`은 공통 베이스 브랜치이므로 절대 AI가 직접 커밋하거나 푸시해서는 안 되며, 반드시 위 접두사를 가진 별도의 브랜치를 생성하여 작업한다.

## 3. 푸시 및 Force Push 금지
- AI는 절대 임의로 `--force`나 `-f` 옵션을 주어 원격 저장소에 강제 푸시를 시도하지 않는다.
- 일반적인 `git push` 역시 사용자의 명시적인 요청이 있을 때만 수행한다.

## 4. 다이어그램 자동 갱신(Drift) 대응
- 푸시 중 `pre-push` 훅에 의해 `docs/graph.dot` 멱등성 검증이 실행된다.
- 훅에 의해 아키텍처 다이어그램이 자동 갱신되어 푸시가 차단(`exit 1`)될 경우, 터미널에 안내된 대로 `git add docs/graph.* && git commit --amend`를 수행하여 갱신된 다이어그램을 커밋에 병합한 뒤 다시 푸시해야 함을 명심한다.
