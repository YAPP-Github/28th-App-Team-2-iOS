# GitHub Pull Request Management Guidelines

리뷰어가 PR 코드를 신속하고 정확하게 리뷰할 수 있도록 PR 생성 시 다음 규칙을 엄격히 준수한다. 모든 PR 조작은 로컬 `gh` CLI 도구를 사용한다.

## 1. PR 생성 규칙 및 메타데이터 바인딩
- **드래프트(Draft) PR 필수화**: 모든 PR은 생성 시 반드시 **`--draft`** 플래그를 추가하여 미완성 상태의 초안으로 등록한다.
- **프로젝트 보드 바인딩 생략 (중복 방지)**:
  - PR 카드를 프로젝트 보드에 직접 추가하면 이슈 카드와 PR 카드가 2중으로 노출되어 보드가 지저분해진다.
  - 따라서 **PR 생성 시에는 `--project` 플래그를 절대 사용하지 않고 보드 연동을 생략한다.** 프로젝트 보드는 오직 '이슈 카드' 단 하나로만 진척 상황을 단일하게 관리한다.
- **담당자 매핑**: 
  - **담당자 지정**: 기본값으로 작업을 수행하는 주체인 **`@me`**를 지정하되, 특수한 지정 요청이 있는 경우 해당 계정명을 지정한다.
  * 명령어 예시: `gh pr create --draft --assignee "@me" --title "[Feat] [운세] 운세 뷰 연동 (#12)" --body-file BODY.md`
- **제목 포맷 (Scoped Type)**: **`[타입] [화면/도메인명] 상세 제목 (#이슈번호)`** 형식을 엄격히 준수한다. (단, 이슈 없이 진행하는 예외적 작업인 경우엔 꼬리의 `(#이슈번호)` 접미사를 생략한다.)
  - **참고**: `[타입]` 및 `[화면/도메인명]`은 단일화된 설정 파일인 [issue-pr-title-conventions.json](../../.github/issue-pr-title-conventions.json)을 준수하여 통일성을 유지하고, 꼬리에 `(#이슈번호)` 만 추가한다. (도메인 생략 조건에 해당하는 설정 작업은 PR 제목에서도 도메인을 생략한다.)
  - **주의**: iOS 전용 저장소이므로, 이슈와 달리 PR 제목에는 맨 앞의 **`[iOS]`** 접두사를 **생략**하고 간결하게 작성한다.
  - 예시: `[Chore] AI 협업 가이드라인(Harness) 구축 (#15)` (도메인 및 [iOS] 생략 적용)
  - 예시: `[Feat] [운세] 운세 화면 UI 구현 (#12)` (도메인 적용, [iOS] 생략 적용)

## 2. PR 본문 템플릿 준수
- PR 작성 시 반드시 `.github/pull_request_template.md` 파일 내용을 기반으로 작성한다.
- 템플릿의 각 섹션(변경 내용, 고민한 점, 테스트 방법 등)을 구체적인 예시와 함께 충실하게 메꾼다.
- UI나 화면 레이아웃의 시각적 변화가 있는 경우, 본문의 `Before / After` 테이블에 스크린샷 또는 화면 녹화본(WebP 애니메이션 또는 비디오)을 반드시 임베딩하여 시각적 확인을 돕는다.
- **이슈 링크 자동화 (Linked Issues)**:
  - 연동된 이슈가 존재하는 경우에 한해, PR 본문 최하단에 **`Closes #이슈번호`** 또는 **`Fixes #이슈번호`** 등의 자동 종료 키워드를 필수로 명시한다. (이슈 없이 진행하는 작업인 경우엔 이 섹션 작성을 생략한다.)

## 3. 라인 단위 셀프 인라인 코멘트(Inline Comment) 작성 규칙 (승인 후 진행)**:
AI는 PR 생성을 완료한 직후, 자의적으로 코멘트를 작성하지 않고 반드시 사용자에게 작성 동의 여부를 가볍게 질문하여 결정한다.
- **질문 템플릿**: `"Draft PR (#PR번호) 생성이 완료되었습니다. 수정한 코드 라인들에 대해 AI 셀프 코드 리뷰 인라인 코멘트를 작성해 둘까요? 아니면 바로 검토하시겠습니까?"`
- **사용자 승인(동의) 시**:
  자신이 수정한 코드 중 구조적으로 중요하거나 로직 설명이 필요한 부분, 혹은 사용자 및 리뷰어와의 조율이 필요한 특정 코드 라인에 대해 **셀프 인라인 코멘트(라인 단위 코멘트)를 작성**한다.
  - **실행 방식 (GitHub API 호출)**:
    `gh api`를 사용하여 해당 PR 번호에 셀프 리뷰 코멘트를 등록한다.
    * **API 엔드포인트**: `POST /repos/{owner}/{repo}/pulls/{pull_number}/reviews`
    * **요청 데이터 포맷**:
      ```json
      {
        "body": "AI 1차 셀프 코드 리뷰 및 설명",
        "event": "COMMENT",
        "comments": [
          {
            "path": "Projects/Feature/FortuneFeature/Sources/FortuneFeature.swift",
            "start_line": 10,
            "start_side": "RIGHT",
            "line": 12,
            "body": "여기에 수정한 이유, 핵심 디자인 패턴 또는 사용자에게 조언/논의할 내용을 적습니다. (단일 라인 작성 시에는 start_line과 start_side는 제외하고 line만 명시 가능)"
          }
        ]
      }
      ```
- **거절 또는 생략 요청 시**: 인라인 코멘트 작성 작업을 생략하고 즉시 대기한다.

## 4. 사용자 권한 위임 및 프로세스 종료
AI는 Draft PR을 올리고 (사용자 승인 시) 1차 셀프 인라인 코멘트 작성을 완료하는 단계까지만 수행한 후 동작을 멈추고 사용자에게 완료 보고를 해야 한다.
- **사용자 전담 영역**:
  - PR 본문 최종 검토 및 추가 피드백 코멘트 작성
  - **Ready for review (드래프트 해제)** 상태 전환
  - 최종 코드 머지(Merge) 진행
- **주의**: AI는 절대로 자의적으로 PR을 머지하거나 드래프트 상태를 해제하지 않는다.
